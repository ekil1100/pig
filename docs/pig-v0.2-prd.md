# Pig v0.2 产品需求文档（PRD）

## 1. 文档信息

- 项目名称：Pig
- 版本：v0.2
- 文档类型：MVP 需求文档
- 产品形态：本地 Coding Agent CLI
- 当前基础：
  - Bun + TypeScript
  - 直接调用 Gemini API
  - 已支持 `list_files`、`read_file`、`run_bash`、`edit_file`
  - 当前更偏 demo/教学实现，缺少可持续使用的交互和安全机制

## 2. 项目背景

Pig 已经具备最小 agent loop：用户输入、模型响应、工具调用、工具结果回传、继续推理。
但当前版本更像“能演示 agent 原理”的样例，还不算一个可以长期使用的 CLI 工具。

要把它变成真正可用的工具，核心问题不在于继续堆功能，而在于补齐以下能力：

1. 会话可持续
2. 文件和命令操作可控
3. 用户能看见 agent 做了什么
4. 中断后可以恢复
5. 在小型代码仓库中能完成真实任务

## 3. 产品定位

Pig v0.2 的定位是：

**一个面向本地代码仓库的轻量级 coding agent CLI。**

它应当支持用户在当前工作目录中，通过自然语言完成以下任务：

- 浏览项目结构
- 阅读文件
- 修改文件
- 执行命令
- 保留会话
- 恢复历史工作
- 在执行高风险操作前请求确认

它不是通用聊天工具，也不是复杂的多 agent 平台。
它的目标是：**在单人、本地、小到中型仓库场景下，成为一个真的能用的开发助手。**

## 4. 产品目标

### 4.1 核心目标

Pig v0.2 需要达到以下目标：

1. 从 demo 升级为可日常使用的 CLI 工具
2. 支持安全、可恢复的 agent 工作流
3. 让用户明确理解 agent 的操作过程和改动结果
4. 在真实代码仓库中完成简单但完整的开发任务
5. 为后续版本的插件系统、更多工具、更强编辑能力打基础

### 4.2 成功标准

满足以下条件，可认为 v0.2 达标：

1. 用户可以在本地仓库里启动 Pig 并连续工作
2. 用户退出后可以恢复会话
3. `run_bash` 执行前会请求确认
4. `edit_file` 写入前会展示改动并请求确认
5. 用户能用 Pig 完成至少 3 类真实任务：
   - 阅读并解释项目文件
   - 修改一个文件
   - 运行一个命令并理解结果
6. README 能指导新用户完成安装和使用

## 5. 非目标

v0.2 不解决以下问题：

1. 多 agent 协作
2. Web UI
3. 云端会话同步
4. 用户登录和权限系统
5. RAG / 向量检索
6. IDE 插件集成
7. 高级 patch engine
8. 多模型路由与自动回退

这些内容可以留到后续版本。

## 6. 目标用户

### 6.1 核心用户

1. 想研究 agent 原理的开发者
2. 想在本地仓库里试用 coding agent 的个人开发者
3. 希望有一个轻量 CLI，而不是复杂 IDE 插件的人

### 6.2 典型用户画像

#### 用户 A：作者本人
- 会写 TypeScript/Bun
- 理解 agent 基本概念
- 希望 Pig 不是一次性 demo，而是能持续演进的工具

#### 用户 B：技术型开发者
- 熟悉命令行
- 愿意让 agent 读文件、改代码、跑命令
- 但要求操作可控、可回溯

## 7. 核心使用场景

### 场景 1：代码理解
用户进入项目目录，启动 Pig，询问：
- “这个仓库主要做什么？”
- “读一下 README 和 package.json，总结启动方式”
- “找出 agent loop 在哪里”

### 场景 2：受控改动
用户要求：
- “把 README 里 setup 部分改得更清楚”
- “把配置文件中的某个字段改掉”

Pig 在改文件前展示改动预览，用户确认后执行。

### 场景 3：命令辅助
用户要求：
- “运行测试并解释错误”
- “看一下 git status”
- “执行构建命令”

Pig 在执行 shell 前显示命令并请求确认。

### 场景 4：中断恢复
用户关闭终端后重新启动 Pig，希望：
- 继续上次对话
- 查看上次做过哪些操作
- 在原来的上下文上继续工作

## 8. MVP 范围

v0.2 的范围只覆盖下面 6 个模块：

1. 会话管理
2. 命令执行确认
3. 文件编辑确认
4. 基础 CLI 命令
5. 项目上下文加载
6. 日志与可观测性

## 9. 功能需求

### 9.1 会话管理

#### 目标
让 Pig 从“一次性聊天”变成“可持续工作”。

#### 功能要求

1. 每次启动 Pig 时自动创建会话
2. 会话内容保存到本地磁盘
3. 会话内容至少包括：
   - 会话 ID
   - 创建时间
   - 当前工作目录
   - 模型名
   - 消息历史
   - 工具调用记录
4. 用户退出后，下次可以恢复最近一次会话
5. 用户可以查看历史会话列表
6. 用户可以显式恢复指定会话

#### CLI 需求

支持以下命令：

- `/history`：显示会话列表
- `/resume`：恢复最近会话或选择某个会话
- `/clear`：清空当前上下文，但不删除历史文件

#### 验收标准

1. 关闭进程后重新启动，历史会话仍然存在
2. 恢复会话后，模型能够继续理解之前的上下文
3. 会话文件结构清晰、可读、可调试

### 9.2 命令执行确认

#### 目标
降低 agent 直接执行 shell 命令带来的风险。

#### 功能要求

1. 当模型调用 `run_bash` 时，CLI 必须先向用户展示完整命令
2. 命令执行前必须等待用户确认
3. 用户至少可以选择：
   - `yes`
   - `no`
4. 命令执行结果需要展示：
   - exit code
   - stdout
   - stderr
   - 超时信息（如有）
5. 命令执行记录应保存进会话日志

#### 范围说明

v0.2 先做“确认执行”，不做复杂权限系统。

#### 验收标准

1. 未经确认，命令不得执行
2. 命令失败时，用户能清楚看到失败原因
3. 超时或输出过长时，CLI 表现稳定

### 9.3 文件编辑确认

#### 目标
让文件写入从“黑盒替换”变成“可审查改动”。

#### 功能要求

1. 当模型调用 `edit_file` 时，CLI 必须展示改动预览
2. 预览至少包括：
   - 文件路径
   - old_string
   - new_string
3. 用户确认后才真正写入文件
4. 当出现以下情况时，需要给出明确错误：
   - 文件不存在但不是创建场景
   - `old_string` 未找到
   - `old_string` 匹配多次
5. 创建文件和更新文件应在提示文案中区分

#### 后续扩展
v0.2 可先用字符串替换预览，后续版本再升级为 unified diff 或 patch 模式。

#### 验收标准

1. 用户在写入前能看见改动内容
2. 多匹配和零匹配时不会静默失败
3. 写入结果会记录到日志中

### 9.4 基础 CLI 命令

#### 目标
让 Pig 的交互从“纯聊天输入框”变成真正的 CLI 工具。

#### 功能要求

支持至少以下命令：

- `/help`：显示命令帮助
- `/tools`：显示当前工具列表
- `/history`：显示会话历史
- `/resume`：恢复会话
- `/clear`：清空当前上下文
- `/exit`：退出程序

#### 行为要求

1. slash commands 不应进入模型上下文
2. slash commands 应由本地 CLI 直接处理
3. 命令输出应简洁、稳定

#### 验收标准

1. 用户可以不依赖 README，直接通过 `/help` 了解主要能力
2. slash commands 不影响模型对话历史

### 9.5 项目上下文加载

#### 目标
减少用户每次都手动告诉 Pig “这个项目是什么”。

#### 功能要求

启动时自动收集当前工作目录的基础上下文，至少包括：

1. 顶层文件和目录列表
2. 若存在，则读取：
   - `README.md`
   - `AGENTS.md`
   - `package.json`
3. 这些信息应进入系统上下文或首轮上下文中
4. 上下文加载失败时不得阻断 CLI 启动

#### 验收标准

1. 在常见仓库中，Pig 启动后对项目基础结构有初步认知
2. 不需要用户一上来就先解释目录结构

### 9.6 日志与可观测性

#### 目标
让用户和开发者都能知道 agent 实际做了什么。

#### 功能要求

1. 每次工具调用都应记录：
   - 工具名
   - 参数
   - 执行时间
   - 结果摘要
2. 会话日志落盘保存
3. CLI 中应输出简洁的执行过程提示，例如：
   - “calling run_bash”
   - “waiting for confirmation”
   - “file updated”

#### 验收标准

1. 事后可以根据日志回溯一轮操作
2. 出现问题时能区分是模型问题、工具问题还是用户拒绝执行

## 10. 非功能需求

### 10.1 可用性

1. 首次启动流程清晰
2. 缺失 API Key 时提示明确
3. 常见错误提示应可读，不要只吐原始异常

### 10.2 可靠性

1. 命令失败不应导致整个会话崩溃
2. 文件编辑失败不应污染会话状态
3. 会话写盘应尽量原子化，避免损坏

### 10.3 可维护性

1. 代码不再集中在单一文件
2. 工具实现应模块化
3. 会话、模型、CLI、工具逻辑分层清晰

### 10.4 安全性

1. shell 命令默认必须确认
2. 文件写入默认必须确认
3. CLI 必须清楚显示即将执行的动作

### 10.5 可移植性

1. 继续优先支持 Bun
2. 兼容常见类 Unix 环境
3. 路径处理应尽量避免平台耦合

## 11. 建议的模块划分

建议重构为以下结构：

```text
src/
  cli/
    commands.ts
    prompt.ts
  core/
    agent-loop.ts
    model.ts
    session.ts
  tools/
    list-files.ts
    read-file.ts
    run-bash.ts
    edit-file.ts
  ui/
    confirm.ts
    render.ts
sessions/
agent.ts
```

#### 模块职责

- `core/agent-loop.ts`
  - 负责对话循环和 function calling 主流程

- `core/model.ts`
  - 封装 Gemini API 请求

- `core/session.ts`
  - 管理会话创建、保存、加载、列出历史

- `tools/*`
  - 单个工具单文件，实现清晰边界

- `cli/*`
  - 处理 slash commands 和用户输入

- `ui/*`
  - 处理确认提示、工具调用显示、日志展示

## 12. 数据设计

### 12.1 会话对象

建议至少包含以下字段：

```ts
type Session = {
  id: string;
  createdAt: string;
  updatedAt: string;
  cwd: string;
  model: string;
  messages: CtxMessage[];
  toolEvents: ToolEvent[];
};
```

### 12.2 工具事件对象

```ts
type ToolEvent = {
  id: string;
  name: string;
  args: Record<string, unknown>;
  status: "confirmed" | "rejected" | "success" | "error";
  startedAt: string;
  finishedAt?: string;
  summary?: string;
};
```

## 13. 用户体验要求

### 13.1 输入输出风格

1. 默认输出应偏简洁
2. 工具调用过程要可见，但不要刷屏
3. 确认问题必须直接，不要含糊

示例：

```text
Pig wants to run:
git status

Proceed? [y/N]
```

```text
Pig wants to edit:
README.md

Old:
Run bun agent.ts

New:
Run bun run agent.ts

Apply? [y/N]
```

### 13.2 错误体验

错误提示要区分类型：

1. 配置错误
   - 缺少 API Key
2. 模型错误
   - API 请求失败
3. 工具错误
   - 文件不存在
   - 命令超时
4. 用户取消
   - command rejected
   - edit rejected

## 14. 验收测试场景

v0.2 发布前，至少要通过以下场景：

### 场景 A：项目理解
输入：
- “这个仓库做什么？”
预期：
- Pig 能读取上下文并给出合理总结

### 场景 B：命令确认
输入：
- “运行 git status”
预期：
- Pig 展示命令并请求确认
- 用户拒绝时不执行

### 场景 C：文件编辑确认
输入：
- “把 README 里的启动命令改掉”
预期：
- Pig 展示修改预览
- 用户确认后写入

### 场景 D：会话恢复
操作：
- 开启会话，对话若干轮后退出
- 重新启动并恢复
预期：
- 能看到历史并继续工作

### 场景 E：异常处理
输入：
- “读一个不存在的文件”
预期：
- 返回明确错误，不崩溃

## 15. 发布范围

v0.2 的交付物应包括：

1. 可运行的 CLI
2. 本地会话保存能力
3. 命令确认能力
4. 文件编辑确认能力
5. slash commands
6. 更新后的 README
7. 基础目录重构

## 16. 风险与注意事项

1. **字符串替换式编辑本身比较脆**
   - v0.2 可以接受，但要把错误提示做好

2. **上下文无限增长**
   - 会话持久化后，历史可能越来越长
   - 未来需要考虑 summary 或截断策略

3. **确认流程会降低速度**
   - 但 v0.2 的优先级是可控，不是极致自动化

4. **单模型绑定**
   - v0.2 不需要解决，但数据结构里最好预留 `model` 字段

## 17. 后续版本方向

v0.3 可以考虑：

1. unified diff / patch 编辑模式
2. 工具权限策略
3. `--cwd`、`--model` 等 CLI 参数
4. 更丰富的上下文加载
5. 插件系统
6. 更好的日志查看体验

## 18. 一句话结论

**Pig v0.2 的核心不是“再加几个工具”，而是把现有 demo 变成一个安全、可恢复、可持续使用的本地 coding agent CLI。**
